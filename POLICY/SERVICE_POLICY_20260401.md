## 1. 문서 개요

### 1.1 목적

본 문서는 올라(Hola) PMS 시스템의 **모든 서비스 정책과 비즈니스 규칙**을 정의한다. 개발, 운영, 인수인계, QA 시 **단일 참조 문서(Single Source of Truth)**로 사용하며, 모든 기능 개발은 본 정책서에 근거하여 진행한다.

### 1.4 기술 용어 해설

본 정책서에서 사용하는 기술 용어를 개발자가 쉽게 이해할 수 있도록 정리한다.

| 용어 | 쉬운 설명 |
|------|----------|
| **Soft Delete** | 데이터를 실제로 DB에서 지우지 않고, `deletedAt` 날짜와 `useYn=false`를 설정하여 "삭제된 것처럼" 처리하는 방식. 조회 시 `@SQLRestriction("deleted_at IS NULL")` 필터가 자동으로 삭제된 데이터를 숨겨준다. 데이터 복구와 감사 추적이 가능하다는 장점이 있다. |
| **멱등성 (Idempotency)** | 같은 요청을 여러 번 보내도 결과가 한 번 보낸 것과 동일한 성질. 예를 들어 부킹엔진에서 `idempotencyKey`를 사용하면, 네트워크 오류로 고객이 "결제" 버튼을 두 번 눌러도 예약이 1건만 생성된다. 두 번째 요청은 첫 번째에서 생성된 기존 예약을 그대로 반환한다. |
| **낙관적 잠금 (Optimistic Lock)** | "충돌이 거의 없을 것"이라고 가정하고, 데이터를 읽을 때는 잠그지 않다가 **저장하는 시점에 버전 번호를 비교**하여 충돌을 감지하는 방식. `@Version` 필드를 사용한다. 예: A와 B가 동시에 같은 예약을 수정하면, 먼저 저장한 쪽은 성공하고 나중에 저장하는 쪽은 "다른 사용자가 이미 수정했습니다" 오류를 받는다. |
| **비관적 잠금 (Pessimistic Lock)** | "충돌이 발생할 가능성이 높다"고 가정하고, 데이터를 읽는 시점에 **DB 행을 잠가서** 다른 트랜잭션이 접근하지 못하게 막는 방식. `SELECT ... FOR UPDATE` SQL을 사용한다. 예: 객실 가용성을 확인하면서 동시에 잠금을 걸어, 두 명이 동시에 같은 객실을 예약하는 것을 원천 차단한다. |
| **TOCTOU (Time Of Check to Time Of Use)** | "확인하는 시점"과 "사용하는 시점" 사이에 데이터가 바뀌는 문제. 예를 들어 "객실 A가 비어있다"고 확인한 후, 실제로 배정하기 전에 다른 요청이 먼저 그 객실을 잡아버리는 상황. 비관적 잠금(`FOR UPDATE`)으로 방지한다. |
| **Upsert** | Update + Insert의 합성어. 이미 존재하면 업데이트하고, 없으면 새로 생성하는 방식. 예: 정산 설정에서 동일한 `(propertyId, countryType)` 조합이 있으면 수정하고, 없으면 신규 생성한다. |
| **전체 교체 방식** | 기존 데이터를 모두 Soft Delete한 후, 새로운 데이터를 처음부터 다시 입력하는 방식. 개별 항목의 추가/수정/삭제를 따로 처리하는 것보다 로직이 단순하다. 예: 취소수수료 정책을 수정하면, 기존 정책 3건을 모두 삭제 처리한 뒤 새 정책 2건을 새로 생성한다. |
| **HMAC (Hash-based Message Authentication Code)** | 비밀키를 이용해 메시지의 위변조를 검증하는 방식. KICC PG 결제에서 `pgCno + amount + transactionDate`를 비밀키로 해시하여 결제 데이터가 중간에 조작되지 않았는지 확인한다. |
| **Greedy 알고리즘** | "현재 시점에서 가장 좋은 선택"을 반복하는 방식. 하우스키핑 자동 배정에서 크레딧이 큰 태스크부터 처리하되, 매번 현재 크레딧이 가장 적은 하우스키퍼에게 배정하여 업무량을 균등하게 분배한다. |
| **폴백 (Fallback)** | 1순위 방법이 실패하거나 없을 때 사용하는 대체 방법. 예: 노쇼 수수료 계산 시 NOSHOW 전용 정책을 먼저 찾고, 없으면 DATE 유형 정책으로 폴백한다. |
| **orphanRemoval** | JPA에서 부모-자식 관계의 자식 엔티티가 부모 컬렉션에서 제거되면 DB에서도 자동 삭제되는 기능. 구역(Section) 매핑 수정 시 `collection.clear()` → `flush()` → `addAll()` 순서로 처리해야 JPA가 올바르게 동작한다. `@Modifying @Query`로 직접 DELETE하면 JPA 캐시와 불일치가 발생하므로 금지. |
| **FOR UPDATE** | 비관적 잠금의 SQL 구문. `SELECT ... FOR UPDATE`로 조회하면 해당 행에 배타적 잠금이 걸려, 현재 트랜잭션이 끝날 때까지 다른 트랜잭션은 해당 행을 수정할 수 없다. |

## 2. 호텔/프로퍼티 관리 정책

### 2.1 호텔 관리

#### 2.1.1 호텔 생성

| 항목 | 정책 |
|------|------|
| 호텔 코드 | 시스템 자동 생성. 형식: `HTL{5자리 순번}` (예: `HTL00001`). DB 시퀀스 기반 |
| 호텔명 | 필수. 최대 200자. **호텔명 중복 불가** (삭제된 호텔 제외) |
| 국가 코드 | 기본값 `+82` (대한민국) |

#### 2.1.2 호텔 삭제

- **하위 프로퍼티가 존재하면 삭제 불가** (HOLA-1002)
- 삭제 시 물리 삭제가 아닌 Soft Delete 처리

### 2.2 프로퍼티 관리

#### 2.2.1 프로퍼티 생성

| 항목 | 정책 |
|------|------|
| 프로퍼티 코드 | 시스템 자동 생성. 형식: `{호텔코드}-P{3자리}` (예: `HTL00001-P001`). Soft Delete 포함 전체 카운트 기반 |
| 프로퍼티명 | 필수. **동일 호텔 내 프로퍼티명 중복 불가** (HOLA-1012) |
| 타임존 | 기본값 `Asia/Seoul` |
| 국가 코드 | 기본값 `+82` |

#### 2.2.2 프로퍼티 삭제

- **하위 데이터(층, 객실, 마켓코드)가 존재하면 삭제 불가** (HOLA-1013)

#### 2.2.3 프로퍼티 운영 설정

| 설정 항목 | 기본값 | 설명 |
|-----------|--------|------|
| 체크인 시간 | `15:00` | 표준 체크인 시각 |
| 체크아웃 시간 | `11:00` | 표준 체크아웃 시각 |
| 세금률 (VAT) | `0%` | 소수점 자릿수 기본 0 |
| 봉사료율 | `0%` | 소수점 자릿수 기본 0 |
| 당일 예약 허용 | `true` | 당일 체크인 예약 가능 여부 |
| 당일 예약 마감 시간 | `18:00` (1080분) | 분 단위. 이 시간 이후 당일 예약 불가 |
| 워크인 오버라이드 | `true` | `true`이면 PMS에서 마감 시간 무시하고 예약 생성 가능 |
| 대실(Dayuse) 허용 | `false` | 활성화 시 대실 예약 가능 |
| 대실 시작 시간 | `10:00` | 대실 이용 가능 시작 시각 |
| 대실 종료 시간 | `20:00` | 대실 이용 가능 종료 시각 |
| 대실 기본 시간 | `5시간` | 대실 기본 이용 시간 |
| 총 객실 수 | `0` | 프로퍼티 전체 객실 수 |

#### 2.2.4 세금/봉사료 계산 정책

| 항목 | 정책 |
|------|------|
| 세금 범위 | 0~100% |
| 봉사료 범위 | 0~100% |
| 소수점 자릿수 | 각각 독립 설정 (0 이상) |
| 반올림 방식 | `ROUND_UP`, `ROUND_DOWN`, `ROUND_FLOOR`, `ROUND_CEILING`, `HALF_UP` 중 선택 |
| 세금 계산 기준 | `(공급가 + 봉사료) * 세금률 / 100` — 봉사료에도 세금 부과 |

> **계산 예시** (공급가 100,000원, 봉사료율 10%, 세금률 10%, 반올림 HALF_UP):
> 1. 봉사료 = 100,000 × 10% = **10,000원**
> 2. 세금 = (100,000 + 10,000) × 10% = **11,000원**
> 3. 총액 = 100,000 + 10,000 + 11,000 = **121,000원**
>
> 주의: 세금은 공급가만이 아닌 **(공급가 + 봉사료)** 합계에 부과된다.

### 2.6 취소수수료 정책 설정

| 항목 | 정책 |
|------|------|
| 저장 방식 | **전체 교체 방식** — 수정 시 기존 정책을 모두 Soft Delete한 뒤, 새로운 정책을 처음부터 다시 입력한다. 개별 항목의 추가/삭제를 따로 추적하지 않아 로직이 단순하다 |
| 기준 유형 | `DATE` (체크인 N일 전) 또는 `NOSHOW` (노쇼 전용) |
| 수수료 유형 | `PERCENTAGE` (1박 요금 대비 %), `FIXED_KRW` (원화 고정), `FIXED_USD` (달러 고정) |
| 수수료 적용 | 소수점 정밀도 15자리, 소수 2자리 |
| 정책 미설정 시 | **무료 취소** (수수료 0원) |

### 2.7 얼리체크인/레이트체크아웃 수수료 정책

| 항목 | 정책 |
|------|------|
| 저장 방식 | **전체 교체 방식** — 수정 시 기존 수수료 정책을 모두 삭제 처리한 뒤, 새로 전달된 정책을 처음부터 다시 입력한다 |
| 정책 유형 | `EARLY_CHECKIN` 또는 `LATE_CHECKOUT` |
| 수수료 유형 | `PERCENT` (기준 객실료 대비 %) 또는 `FIXED` (고정 금액) |
| 시간 범위 | `timeFrom` ~ `timeTo`. **시작 시간이 종료 시간보다 늦으면 오류** (HOLA-4051) |
| 빈 목록 전송 시 | 기존 정책 전체 삭제 (수수료 미부과 전환) |

### 2.8 프로퍼티 정산 설정

| 항목 | 정책 |
|------|------|
| 저장 방식 | **Upsert 방식** — 같은 프로퍼티+국가 유형 조합의 정산 설정이 이미 있으면 기존 데이터를 수정하고, 없으면 새로 생성한다 |
| 유니크 제약 | `(property_id, country_type)` |

### 2.9 프로퍼티 이미지 관리

| 항목 | 정책 |

## 5. 요금(레이트) 관리 정책

### 5.1 레이트코드(Rate Code)

#### 5.1.1 생성 규칙

| 항목 | 정책 |
|------|------|
| 레이트코드 | 필수. 최대 50자. **동일 프로퍼티 내 중복 불가** (HOLA-3001) |
| 레이트명(한글) | 필수. 최대 200자 |
| 매출 카테고리 | 필수 |
| 통화 | 필수 |
| 판매 시작/종료일 | 필수. **종료일이 시작일보다 빠르면 오류** (HOLA-3004) |
| 최소 숙박일 | 필수. 최소 1일 |
| 최대 숙박일 | 필수. 최소 1일. **최대 < 최소이면 오류** (HOLA-3003) |
| 숙박 유형 | `OVERNIGHT` (기본) 또는 `DAY_USE`. null/빈값은 OVERNIGHT 자동 전환 |
| 객실 타입 매핑 | 수정 시 **전체 교체** |
| 유료 서비스 매핑 | 수정 시 **전체 교체** |
| 삭제 시 | 객실 타입 매핑 + 유료 서비스 매핑 정리 후 Soft Delete |

#### 5.1.2 유효 레이트코드 판별 규칙

예약/부킹 시 레이트코드 사용 가능 여부 판별:

1. `useYn = true` (활성 상태)
2. 프로퍼티에서 대실 비활성화 시 → `DAY_USE` 레이트 제외
3. `saleStartDate`, `saleEndDate` 모두 설정되어 있을 것
4. 체크인 날짜가 판매 기간 내에 포함 (inclusive)
5. 숙박일수가 `minStayDays` 이상, `maxStayDays` 이하 (maxStayDays > 0일 때)
6. `DAY_USE` 레이트는 가격 커버리지 검증 건너뜀 (DayUseRate 테이블 사용)
7. `OVERNIGHT` 레이트는 체크인~체크아웃 전일 모든 날짜에 요금 커버리지 필수

### 5.2 요금 단가(Rate Pricing)

#### 5.2.1 저장 방식

- **전체 교체 방식**: 수정 시 기존 단가와 인원별 추가요금을 모두 Soft Delete한 뒤, 새로 전달된 데이터를 처음부터 다시 입력한다. 개별 행의 변경/삭제를 추적하지 않아 로직이 단순하다

#### 5.2.2 기간 검증 규칙

| 규칙 | 상세 |
|------|------|
| 필수 값 | 시작일, 종료일 모두 필수 (HOLA-3005) |
| 기간 유효성 | 종료일 >= 시작일 (HOLA-3006) |
| 판매 기간 내 | 단가 기간이 레이트코드 판매 기간 내에 포함 (HOLA-3007) |
| 중복 검증 | 두 단가 행의 날짜 범위가 겹치고 AND 요일 플래그가 겹치면 **중복 오류** (HOLA-3008) |

#### 5.2.3 요일별 가격 적용

- 월~일 7개 요일 플래그 (기본 모두 `true`)
- 특정 요일만 적용하는 요금 설정 가능 (예: 금-토만 성수기 요금)

#### 5.2.4 인원별 추가요금 (Rate Pricing Person)

| 항목 | 정책 |
|------|------|
| 유니크 제약 | `(rate_pricing_id, person_type, person_seq)` |
| 인원 유형 | `ADULT`, `CHILD` |
| 성인 추가 | 2번째 성인부터 추가요금 적용 (seq 2, 3, ...) |
| 어린이 추가 | 1번째 어린이부터 추가요금 적용 (seq 1, 2, ...) |
| 저장 조건 | `supplyPrice > 0`인 건만 저장 (0원 추가요금은 스킵) |

### 5.3 가격 계산 엔진

#### 5.3.1 1박당 요금 계산 공식

```
1. 날짜 범위 + 요일 매칭으로 RatePricing 조회
2. supplyPrice = baseSupplyPrice + 인원별 추가요금 합계
3. serviceCharge = supplyPrice * serviceChargeRate / 100
4. tax = (supplyPrice + serviceCharge) * taxRate / 100
5. total = supplyPrice + serviceCharge + tax
```

> **1박 요금 계산 예시** (성인 2명, 어린이 1명 / 기본 공급가 80,000원 / 봉사료 10% / 세금 10%):
> 1. RatePricing 조회: 체크인 날짜 + 요일로 매칭되는 단가 행을 찾는다
> 2. 인원별 추가요금: 성인 2번째 +20,000원, 어린이 1번째 +10,000원
> 3. supplyPrice = 80,000 + 20,000 + 10,000 = **110,000원**
> 4. serviceCharge = 110,000 × 10 / 100 = **11,000원**
> 5. tax = (110,000 + 11,000) × 10 / 100 = **12,100원**
> 6. total = 110,000 + 11,000 + 12,100 = **133,100원** (1박)

#### 5.3.2 요금 매칭 우선순위

- 동일 날짜+요일에 복수 RatePricing 매칭 시 → **가장 좁은 날짜 범위** 우선 (가장 구체적인 가격 적용)

#### 5.3.3 가격 커버리지 검증

- 체크인일부터 체크아웃 전일까지 **모든 날짜에 매칭되는 단가가 존재**해야 함
- 날짜 범위 + 요일 플래그 모두 일치해야 커버리지 충족
- 누락 날짜 1개라도 있으면 `RESERVATION_RATE_NOT_APPLICABLE` (HOLA-4020)

### 5.4 할인/할증 설정 (Down/Up Sale)

| 항목 | 정책 |
|------|------|
| 부호 | `+` (할증) 또는 `-` (할인) |
| 단위 | `PERCENT` (%) 또는 `AMOUNT` (고정금액) |
| 적용 | 기본 공급가에 대해 적용 |

### 5.5 프로모션코드

| 항목 | 정책 |
|------|------|
| 코드 | **동일 프로퍼티 내 중복 불가** (HOLA-3501) |
| 연결 | 반드시 하나의 레이트코드에 연결 (`rateCodeId` 필수) |
| 유형 | `COMPANY`, `PROMOTION`, `OTA` |
| 기간 | 시작일, 종료일 필수. **종료일 < 시작일이면 오류** (HOLA-3502) |
| 할인/할증 | 레이트 단가와 동일한 Down/Up Sale 구조 적용 |
| 삭제 조건 | **활성 상태에서는 삭제 불가** (HOLA-1004). 먼저 비활성화 필요 |

## 6. 예약 관리 정책

### 6.1 예약 번호 체계

| 구분 | 형식 | 예시 |
|------|------|------|
| 마스터 예약번호 | `{프로퍼티코드}{YYMMDD}-{4자리 일일순번}` | `GMP260308-0001` |
| 서브 예약번호 | `{마스터번호}-{2자리 Leg순번}` | `GMP260308-0001-01` |
| 컨펌번호 | 8자리 영숫자 UUID. 충돌 시 3회 재시도 → 12자리 폴백 | `A3F8K2M1` |

- 순번 생성 동시성 보호: 비관적 잠금(`FOR UPDATE`)으로 현재 최대 순번을 조회 → +1 증가 → INSERT. 만약 두 요청이 거의 동시에 들어와 잠금을 놓치면, INSERT 시 유니크 제약 위반이 발생하고 순번을 +1 재시도하여 충돌을 해결한다

### 6.2 예약 상태 체계

#### 6.2.1 서브 예약(Leg) 상태

| 상태 | 코드 | 설명 |
|------|------|------|
| 예약 확정 | `RESERVED` | 예약 생성 완료 |
| 체크인 | `CHECK_IN` | 레거시 호환. INHOUSE와 동일 취급 |
| 투숙 중 | `INHOUSE` | 체크인 완료, 재실 중 |
| 체크아웃 | `CHECKED_OUT` | 퇴실 완료 (종료 상태) |
| 취소 | `CANCELED` | 예약 취소 (종료 상태) |
| 노쇼 | `NO_SHOW` | 미도착 처리 (종료 상태) |

#### 6.2.2 상태 전이 매트릭스

```
RESERVED   → INHOUSE, CANCELED, NO_SHOW
CHECK_IN   → INHOUSE, CANCELED            (레거시 호환)
INHOUSE    → CHECKED_OUT
CHECKED_OUT → (종료 - 전이 불가)
CANCELED    → (종료 - 전이 불가)
NO_SHOW     → (종료 - 전이 불가)
```

#### 6.2.3 종료 상태 제약

- `CHECKED_OUT`, `CANCELED`, `NO_SHOW` 상태의 예약은 **수정 불가**

#### 6.2.4 마스터 예약 상태 자동 결정

마스터 상태는 하위 서브 예약 상태로부터 **자동 파생**:

| 조건 | 마스터 상태 |
|------|------------|
| 모든 활성 Leg이 CHECKED_OUT | `CHECKED_OUT` |
| 활성 Leg 중 하나라도 INHOUSE 또는 CHECK_IN | `INHOUSE` |
| 모든 Leg이 CANCELED | `CANCELED` |
| 모든 Leg이 NO_SHOW (CANCELED 제외) | `NO_SHOW` |
| 그 외 | `RESERVED` |

> "활성 Leg" = CANCELED, NO_SHOW가 아닌 서브 예약

### 6.6 예약 취소 정책

#### 6.6.1 취소 가능 조건

| 조건 | 정책 |
|------|------|
| 취소 가능 상태 | `RESERVED` 또는 `CHECK_IN(INHOUSE)` |
| 미결제 잔액 | `grandTotal > netPaid`이면 **취소 불가** (HOLA-4084) — 먼저 결제 조정 필요 |
| 취소수수료 > 결제액 | 수수료가 지불 금액을 초과하면 **취소 불가** (HOLA-4083) |

#### 6.6.2 취소 수수료 매칭 알고리즘

1. `잔여일 = 체크인 - 오늘` (음수면 0)
2. DATE 기준 정책을 `daysBefore ASC` 정렬
3. `잔여일 <= daysBefore`인 첫 번째 정책 선택
4. 매칭 정책 없음 → **무료 취소** (수수료 0원)
5. 정책 미설정 → **무료 취소**

> **취소수수료 매칭 예시** (체크인: 4월 10일, 오늘: 4월 8일):
> - 잔여일 = 4월 10일 - 4월 8일 = **2일**
> - 정책 목록 (daysBefore ASC): `[{1일, 100%}, {3일, 50%}, {7일, 30%}]`
> - 매칭: 잔여일(2) <= 3일인 첫 번째 정책 → **50% 적용**
>   - (1일 정책은 건너뜀: 잔여일 2 > 1이므로 조건 불충족)
>   - (3일 정책이 첫 매칭: 잔여일 2 <= 3이므로 선택)
>
> **당일 취소 예시** (체크인: 4월 8일, 오늘: 4월 8일):
> - 잔여일 = 0일 → `0 <= 1` → **1일 정책(100%) 적용**

#### 6.6.3 수수료 계산

| 수수료 유형 | 계산 방식 |
|------------|----------|
| `PERCENTAGE` | 첫째 날 요금 * feeAmount / 100 (HALF_UP 반올림, 소수점 0자리) |
| `FIXED_KRW` | 고정 원화 금액 |
| `FIXED_USD` | 미지원 (수수료 0원 처리) |

- **첫째 날 요금 기준**: `originalFirstNightTotal` 우선 (객실 업그레이드 후에도 원래 가격 기준). 없으면 첫 서브예약의 첫 DailyCharge 합계

#### 6.6.4 취소 처리 흐름

1. 환불액 = 순결제액 - 취소수수료 (최소 0원)
2. OTA 예약: PG 환불액을 0으로 설정 (OTA 채널에서 환불 처리). 메모 기록
3. 결제 상태 → `REFUNDED`. `grandTotal` 보존 (감사 추적용)
4. 서비스 아이템 재고 반환
5. 마스터 + 전체 서브 → `CANCELED`

#### 6.6.5 노쇼 수수료 정책

| 항목 | 정책 |
|------|------|
| 노쇼 처리 가능 시점 | 체크인 날짜 <= 오늘 (미래 예약에 대해 노쇼 불가) |
| 수수료 정책 | `NOSHOW` 유형 정책 우선 적용. 없으면 DATE 유형 폴백 |
| Leg별 노쇼 | 개별 Leg 단위로 노쇼 처리 가능. Leg별 수수료 계산 + 환불 |

#### 6.7.5 상태 변경 범위

| 변경 유형 | 동작 |
|-----------|------|
| Leg 단위 | 지정된 서브만 상태 변경 |
| 벌크 (NO_SHOW/CANCELED) | 전이 가능한 모든 Leg에 적용 |
| 벌크 (INHOUSE/CHECKED_OUT) | 전이 유효한 Leg에만 적용 |
| 마지막 Leg 체크아웃 | 미결제 잔액 검증 수행 |

### 6.8 객실 배정 정책

#### 6.8.1 가용성 검증 단계

| 단계 | 검증 내용 |
|------|----------|
| L1 (객실 단위) | 특정 객실에 기간 겹침 없는지 확인. `checkIn < 기존.checkOut AND checkOut > 기존.checkIn` |
| L2 (타입 단위) | 타입별 가용 = 등록 객실 수 - 활성 예약 수 |
| L3 (오버부킹) | L2 가용 <= 0이면 경고 (관리자 판단) |

#### 6.8.2 객실 상태 제한

- **배정 불가 상태**: `OOO`, `OOS`, `DIRTY`, `PICKUP`
- `CLEAN` 또는 `INSPECTED` 상태만 배정 가능

#### 6.8.3 OOO/OOS 기간 검증

- 객실 배정 시 해당 기간에 겹치는 OOO/OOS가 있으면 배정 불가 (HOLA-5005)

#### 6.8.4 동시성 보호

- **비관적 잠금으로 동시 예약 충돌 방지**: 객실 가용성을 확인하는 시점에 `SELECT ... FOR UPDATE`로 해당 행을 잠근다. 이렇게 하면 "A가 빈 객실을 확인하는 사이에 B가 먼저 예약해버리는" 상황(TOCTOU 문제)을 원천 차단한다.
- 개별 객실 배정 시에도, 타입 단위 가용 수 조회 시에도 모두 비관적 잠금 적용

#### 6.8.5 객실 정렬 규칙

1. 가용 객실 우선
2. 객실 번호 순
3. 추천 타입(현재 레이트코드 매핑) 그룹 우선, 그 다음 타입 코드 순

#### 6.8.6 업그레이드 시 레이트코드 선택 우선순위

1. 현재 레이트코드가 대상 객실 타입에 매핑되어 있으면 그대로 사용
2. 대상 객실 타입에 매핑된 첫 번째 활성 레이트코드
3. 현재 레이트코드 폴백

## 8. 결제/정산 정책

### 8.1 결제 상태 체계

| 상태 | 코드 | 판별 기준 |
|------|------|----------|
| 미결제 | `UNPAID` | `netPaid <= 0` AND `grandTotal > 0` |
| 부분결제 | `PARTIAL` | `0 < netPaid < grandTotal` |
| 완납 | `PAID` | `netPaid == grandTotal` (또는 `grandTotal <= 0`이면서 결제 없음) |
| 초과결제 | `OVERPAID` | `netPaid > grandTotal` |
| 환불완료 | `REFUNDED` | 취소/노쇼 처리 시 명시적 설정. **자동 재계산으로 덮어쓰지 않음** |

- `netPaid = totalPaidAmount - refundAmount - cancelFeeAmount`

> **결제 상태 판별 예시** (총 청구액 200,000원):
> - 결제 0원 → netPaid=0, grandTotal>0 → **UNPAID**
> - 결제 100,000원 → netPaid=100,000 < 200,000 → **PARTIAL**
> - 결제 200,000원 → netPaid=200,000 = 200,000 → **PAID**
> - 결제 200,000원 후 조정 -50,000원 → grandTotal=150,000, netPaid=200,000 > 150,000 → **OVERPAID**
> - 취소 처리 시 → 환불 후 명시적으로 **REFUNDED** 설정 (자동 재계산 대상 아님)

### 8.2 결제 처리 규칙

| 항목 | 정책 |
|------|------|
| 결제 불가 상태 | `CHECKED_OUT`, `CANCELED`, `NO_SHOW` |
| 결제 금액 | 0보다 커야 함 (HOLA-4028) |
| 최대 결제 | 잔여 금액 초과 불가 (HOLA-4026) |
| 미지정 금액 | 금액 미지정 시 전체 잔여 금액 자동 부과 |
| 단일 Leg | `subReservationId` 미지정 시 자동 할당 |
| 잔여 금액 0 | 이미 완납이면 결제 불가 (HOLA-4021) |
| 동시성 제어 | `@Version` 낙관적 잠금 — 두 관리자가 동시에 같은 예약의 결제를 처리하면, 나중에 저장하는 쪽에 충돌 오류 발생 (HOLA-4027) |

### 8.3 트랜잭션 유형

| 유형 | 코드 | 설명 |
|------|------|------|
| 결제 | `PAYMENT` | 일반 결제 (카드/현금) |
| 환불 | `REFUND` | 환불 트랜잭션 |

### 8.9 환불 처리 정책

#### 8.9.1 환불 분배 로직

1. **PG 결제 우선 환불**: PG 환불 가능액 = PG 결제 총액 - 이미 PG 환불된 금액
2. 실제 PG 환불 = `min(환불금액, PG 환불 가능액)`
3. 나머지 = 비PG 환불 (`MANUAL_CONFIRMED` 처리)

> **환불 분배 예시** (카드 150,000원 + 현금 50,000원 결제, 취소수수료 30,000원):
> 1. 총 결제액 = 200,000원, 환불액 = 200,000 - 30,000 = **170,000원**
> 2. PG 환불 가능액 = PG 결제(150,000) - 이미 PG 환불(0) = **150,000원**
> 3. 실제 PG 환불 = min(170,000, 150,000) = **150,000원** → KICC API 호출
> 4. 나머지 = 170,000 - 150,000 = **20,000원** → 현금 환불 (MANUAL_CONFIRMED)
> 5. 현금으로 결제한 50,000원 중 20,000원만 환불, 나머지 30,000원은 취소수수료에 해당

#### 8.9.2 PG 환불

| 항목 | 정책 |
|------|------|
| 전체/부분 | 취소수수료 0 + 비PG 나머지 없음 → 전체 취소 / 그 외 부분 취소 |

## 10. 하우스키핑 정책

### 10.1 태스크 상태 체계

| 상태 | 코드 | 설명 |
|------|------|------|
| 대기 | `PENDING` | 생성됨, 작업 시작 전 |
| 진행중 | `IN_PROGRESS` | 청소 진행 중 |
| 일시정지 | `PAUSED` | 일시 중단 |
| 완료 | `COMPLETED` | 청소 완료 |
| 검수완료 | `INSPECTED` | 감독자 검수 완료 |
| 취소 | `CANCELLED` | 취소됨 |

#### 10.1.1 상태 전이 매트릭스

```
PENDING     → IN_PROGRESS, CANCELLED
IN_PROGRESS → PAUSED, COMPLETED, CANCELLED
PAUSED      → IN_PROGRESS, CANCELLED
COMPLETED   → INSPECTED
INSPECTED   → (종료)
CANCELLED   → (종료)
```

#### 10.1.2 취소 제한

- `COMPLETED`, `INSPECTED`, `CANCELLED` 상태에서는 **취소 불가** (이미 객실 상태 변경됨)

### 10.2 태스크 유형 및 크레딧

| 유형 | 코드 | 기본 크레딧 | 설명 |
|------|------|------------|------|
| 체크아웃 청소 | `CHECKOUT` | 1.0 | 퇴실 후 전면 청소 |
| 재실 청소 | `STAYOVER` | 0.5 | 투숙 중 일상 청소 |
| 턴다운 서비스 | `TURNDOWN` | 0.3 | 저녁 침대 정리 |
| 정밀 청소 | `DEEP_CLEAN` | 2.0 | 비수기 정밀 청소 |
| 간단 정리 | `TOUCH_UP` | 0.3 | 간단 마무리 |

- 크레딧 정밀도: 소수점 1자리
- 미지정 유형: 기본 크레딧 1.0

### 10.3 태스크 우선순위

| 우선순위 | 코드 |
|----------|------|
| 긴급 | `RUSH` |
| 높음 | `HIGH` |
| 보통 | `NORMAL` (기본값) |
| 낮음 | `LOW` |

#### 10.3.1 긴급(Rush) 자동 감지

| 항목 | 정책 |
|------|------|
| 기준 | `nextCheckinAt` 설정 시, 현재 ~ 체크인까지 남은 시간 |
| 임계값 | `rushThresholdMinutes` (기본 120분) |
| 적용 | 남은 시간이 0~임계값 이내면 자동 RUSH 승격 |
| 제외 | 이미 RUSH인 태스크, `nextCheckinAt` 미설정 |

### 10.4 태스크 생성 정책

#### 10.4.1 수동 생성

| 항목 | 정책 |
|------|------|
| 객실 검증 | 존재하는 객실이어야 함 |
| 중복 방지 | 동일 객실 + 동일 날짜에 활성 태스크 1개만 허용 |
| 크레딧 | 미지정 시 태스크 유형별 기본 크레딧 자동 적용 |
| 배정 | 생성 시 `assignedTo` 지정 가능 |

#### 10.4.2 체크아웃 자동 생성

| 항목 | 정책 |
|------|------|
| 조건 | `autoCreateCheckout = true` (기본값) |
| 시점 | 체크아웃 이벤트 발생 시 |
| 유형 | `CHECKOUT`, 우선순위 `NORMAL` |
| 중복 | 이미 활성 태스크 존재 시 생성 안 함 |
| Rush | 자동 감지 적용 |

#### 10.4.3 일일 자동 생성 (배치)

| 대상 | 조건 | 태스크 유형 |
|------|------|-----------|
| VD (빈방+더러움) | `foStatus=VACANT`, `hkStatus=DIRTY` | `CHECKOUT` |
| OD (재실+더러움) | `foStatus=OCCUPIED`, `hkStatus=DIRTY` | `STAYOVER` |

- 이미 활성 태스크 있는 객실은 건너뜀
- VD: Rush 자동 감지 적용. OD: Rush 미적용

### 10.5 태스크 배정 정책

#### 10.5.1 수동 배정

| 항목 | 정책 |
|------|------|
| 배정 기록 | `assignedTo`, `assignedBy` (현재 사용자), `assignedAt` 타임스탬프 |
| 배정 해제 | 모든 배정 필드 초기화 |
| 일괄 배정 | 태스크 ID 목록 + 담당자. 담당자 null이면 배정 해제 |

#### 10.5.2 자동 배정 알고리즘

1. **가용 인력 조회**: 출근부 `isAvailable=true` 기록 조회. 출근부 미생성 시 전체 활성 하우스키퍼
2. **미배정 태스크**: 당일 `assignedTo = null` + 취소/검수완료 제외
3. **태스크 미존재 시**: `generateDailyTasks()` 호출 → 재조회
4. **구역 매핑**: 층 → 구역 → 해당 구역 하우스키퍼 조회
5. **크레딧 균형 배정**: 업무량을 최대한 균등하게 분배하는 알고리즘
   - 크레딧이 큰 태스크(예: 정밀청소 2.0)부터 먼저 배정한다
   - 매번 "현재 배정된 크레딧 합계가 가장 적은" 하우스키퍼에게 배정한다
   - 구역 매핑이 있으면 → 해당 구역 담당 하우스키퍼 중에서 선택
   - 구역 매핑이 없으면 → 전체 가용 하우스키퍼 중에서 선택
   
> **배정 예시** (하우스키퍼 A/B, 태스크 3건: 정밀청소 2.0, 체크아웃 1.0, 재실 0.5):
> 1. 정밀청소(2.0) → A(0) vs B(0) → **A에게** (A 누적: 2.0)
> 2. 체크아웃(1.0) → A(2.0) vs B(0) → **B에게** (B 누적: 1.0)
> 3. 재실(0.5) → A(2.0) vs B(1.0) → **B에게** (B 누적: 1.5)
> → 결과: A=2.0, B=1.5 (최대한 균등하게 분배됨)

#### 10.5.3 태스크시트 재분배

- 2개 이상 시트 + 배정된 하우스키퍼 필요
- `PENDING` 또는 `PAUSED` 태스크만 대상 (진행/완료는 유지)
- 크레딧 내림차순 정렬 → 최소 크레딧 시트에 배정 (Greedy)

| 관리자 직접 | `APPROVED` | PMS에서 관리자가 직접 등록 |

| 상태 | 코드 |
|------|------|
| 대기 | `PENDING` |

#### 10.9.3 휴무 생성 규칙

| 규칙 | 정책 |
|------|------|

- 해당 날짜에 `WORKING`/`LEFT` 상태면 **승인 불가** (HOLA-8071)

- **본인 것만** 취소 가능. 타인 것 취소 시 `HK_TASK_ACCESS_DENIED` (HOLA-8030)

### 10.11 청소 정책(Cleaning Policy)

#### 10.11.1 2단계 오버라이드 시스템

| 단계 | 우선순위 | 설명 |
|------|---------|------|
| Layer 1 | 기본 | 프로퍼티 `HkConfig` 전체 설정 |
| Layer 2 | 우선 | 객실타입별 `HkCleaningPolicy` (null 필드는 Layer 1 폴백) |

> **2단계 오버라이드 동작 예시**:
> - 프로퍼티 전체 설정(Layer 1): `stayoverEnabled=true`, `stayoverCredit=0.5`, `dndPolicy=SKIP`
> - 디럭스룸 타입 정책(Layer 2): `stayoverCredit=0.8`, `dndPolicy=null`
> - **최종 적용값**: `stayoverEnabled=true`(Layer 1), `stayoverCredit=0.8`(Layer 2 우선), `dndPolicy=SKIP`(Layer 2가 null이므로 Layer 1 사용)
>
> 즉, 객실타입별 정책에 값이 있으면 그 값을 쓰고, 없으면(null) 프로퍼티 전체 설정값을 사용한다.

#### 10.11.2 정책 설정 항목

| 설정 | 기본값 | 설명 |
|------|--------|------|
| stayoverEnabled | `false` | 재실 청소 활성화 |
| stayoverFrequency | `1` | 일일 재실 청소 횟수 |
| turndownEnabled | `false` | 턴다운 서비스 활성화 |
| stayoverCredit | `0.5` | 재실 청소 크레딧 |
| turndownCredit | `0.3` | 턴다운 크레딧 |
| stayoverPriority | `NORMAL` | 재실 청소 우선순위 |
| dndPolicy | `SKIP` | DND 처리 정책 |

### 10.12 DND(Do Not Disturb) 처리 정책

| 정책 | 코드 | 동작 |
|------|------|------|
| 건너뛰기 | `SKIP` | 해당 객실 무시. DND 연속일 카운트만 계속 증가. 투숙객이 DND를 해제하기 전까지 청소 태스크를 생성하지 않는다 |
| 오후 재시도 | `RETRY_AFTERNOON` | 오전에 DND인 객실에 대해, 해당 객실에 활성 태스크가 없으면 오후 14:00에 재시도 청소 태스크를 생성한다. `dndSkipped=true`로 마킹하여 오전에 한번 건너뛰었음을 기록한다 |
| N일 후 강제 | `FORCE_AFTER_DAYS` | DND가 설정된 지 `dndMaxSkipDays`일(기본 3일) 이상 지속되면, DND를 강제 해제하고 HIGH 우선순위 청소 태스크를 오전 10:00에 생성한다. 아직 최대 일수 미달이면 SKIP과 동일하게 동작한다 |

> **FORCE_AFTER_DAYS 동작 예시** (dndMaxSkipDays=3):
> - 1일차: DND 설정됨, consecutiveDndDays=1 → SKIP (청소 안 함)
> - 2일차: consecutiveDndDays=2 → SKIP
> - 3일차: consecutiveDndDays=3 → **강제 해제** + HIGH 우선순위 청소 태스크 생성

### 10.13 스케줄러(배치) 정책

#### 10.13.1 실행 주기

- **1분마다** 실행 (`0 * * * * *`)
- 프로퍼티별 독립 처리. 1개 프로퍼티 오류 시 다른 프로퍼티 영향 없음

#### 10.13.2 일일 이벤트 (프로퍼티별)

| 이벤트 | 기본 시간 | 설정 키 | 동작 |
|--------|----------|---------|------|
| OC→OD 전환 | `05:00` | `odTransitionTime` | OC 객실 → DIRTY 전환 + DND 연속일 증가 |
| 일일 태스크 생성 | `06:00` | `dailyTaskGenTime` | ① 재실 청소 생성 ② VD/OD 태스크 생성 ③ DND 처리 |

#### 10.13.3 SecurityContext 주의사항

- 스케줄러(`@Scheduled`)는 사용자 로그인 없이 시스템이 자동 실행하므로, "현재 로그인한 사용자" 정보가 없다
- 따라서 `accessControlService.validatePropertyAccess()`처럼 "현재 사용자가 이 프로퍼티에 접근 권한이 있는지" 확인하는 메서드를 호출하면 런타임 오류가 발생한다
- **해결 방법**: 스케줄러에서 호출하는 서비스 메서드는 접근 권한 검증을 건너뛰는 별도의 내부 전용 메서드로 분리한다

## 12. 데이터 관리 정책

### 12.1 Soft Delete 원칙

| 항목 | 정책 |
|------|------|
| 원칙 | **물리 삭제(DB에서 실제로 지우는 것) 절대 금지**. 모든 삭제는 Soft Delete로 처리 |
| 메커니즘 | `deletedAt`에 현재 시간을 기록하고 `useYn`을 `false`로 설정하여 "삭제된 것처럼" 처리한다. 실제 DB 행은 그대로 유지되어 감사 추적 및 데이터 복구가 가능하다 |
| 자동 필터링 | `@SQLRestriction("deleted_at IS NULL")`이 모든 엔티티에 적용되어 있어, 일반 조회 시 삭제된 데이터가 자동으로 제외된다. 개발자가 매 쿼리마다 삭제 여부를 체크할 필요가 없다 |
| 예외 | `HkTaskLog` — 감사 로그 테이블이므로 Soft Delete도 하지 않음. 한번 기록된 로그는 영구 보존 |

### 12.2 감사(Auditing)

| 필드 | 자동 설정 |
|------|----------|
| `createdAt` | 생성 시 현재 시간 (수정 불가) |
| `updatedAt` | 수정 시 현재 시간 |
| `createdBy` | 현재 로그인한 사용자의 ID. 스케줄러 등 로그인 없이 실행되는 경우 `SYSTEM` |
| `updatedBy` | 현재 로그인한 사용자의 ID. 스케줄러 등 로그인 없이 실행되는 경우 `SYSTEM` |

### 12.3 낙관적 잠금 (버전 비교 방식)

`@Version` 필드를 사용하여 데이터 저장 시점에 충돌을 감지한다. 읽을 때는 잠금을 걸지 않으므로 성능이 좋고, 충돌이 드문 경우에 적합하다. 충돌 발생 시 `OptimisticLockException`이 발생하며 사용자에게 재시도를 안내한다.

| 대상 | 사유 | 충돌 시 사용자 메시지 |
|------|------|---------------------|
| MasterReservation | 두 관리자가 동시에 같은 예약을 수정하는 것 방지 | "다른 사용자가 이미 수정했습니다" |
| SubReservation | 같은 Leg에 대한 동시 수정 방지 | 동일 |
| ReservationPayment | 같은 예약에 대한 동시 결제 처리 방지 (HOLA-4027) | "결제 처리 중 충돌이 발생했습니다" |
| HkDailyAttendance | 같은 하우스키퍼의 출퇴근 동시 처리 방지 | 동일 |

### 12.4 비관적 잠금 (DB 행 잠금 방식)

`SELECT ... FOR UPDATE`로 데이터를 읽는 시점에 DB 행을 잠근다. 충돌이 빈번하거나, 충돌 시 재시도가 어려운 경우에 사용한다. 잠금이 해제될 때까지 다른 트랜잭션은 대기한다.

| 대상 | 사유 | 왜 비관적 잠금인가 |
|------|------|-------------------|
| 예약번호 순번 생성 | 일일 순번 고유성 보장 | 순번 중복 시 복구가 어려우므로, 확인 시점에 잠가서 원천 차단 |
| 객실 가용성 검증 | 동시 예약 시 같은 객실 이중 배정 방지 | "빈 객실 확인→배정" 사이 틈을 없애야 함 |
| 타입별 가용성 조회 | 부킹엔진 동시 접수 시 오버부킹 방지 | 외부 고객 대상이라 충돌 빈도가 높음 |
| 재고 예약 | 동시 서비스 예약 시 재고 초과 방지 | 재고 부족은 결제 후 복구가 어려움 |

## 13. 대시보드 정책

### 13.1 대시보드 접근 권한

| 조회 범위 | 허용 역할 | 설명 |
|-----------|----------|------|
| 전체 프로퍼티 랭킹/요약 | `SUPER_ADMIN` | 전체 호텔/프로퍼티의 통합 KPI 조회 |
| 프로퍼티별 KPI | 모든 인증된 사용자 | 선택한 프로퍼티의 개별 KPI 조회 |

### 13.2 KPI 지표 계산 기준

| 지표 | 계산 방식 | 비고 |
|------|----------|------|
| **OCC% (객실 점유율)** | (판매 객실 수 / 총 판매 가능 객실 수) × 100 | **숙박(OVERNIGHT) 예약만** 포함. 대실(Dayuse)은 제외 |
| **ADR (평균 객실 단가)** | 총 객실 매출 / 판매 객실 수 | 숙박 객실만 기준 |
| **RevPAR (가용 객실당 매출)** | 총 객실 매출 / 총 판매 가능 객실 수 | 또는 OCC% × ADR |
| **대실 매출** | 대실(Dayuse) 예약의 매출 합계 | 숙박 KPI와 별도 항목으로 표시 |

### 13.3 트렌드 비교

| 항목 | 정책 |
|------|------|
| 전일 대비 | 어제의 OCC%, ADR, RevPAR을 함께 계산하여 등락 추이를 표시한다 |
| 7일 픽업 추이 | 오늘부터 향후 7일간의 예약 건수 + 매출 합계를 일별로 집계하여 추이를 표시한다 |

### 13.4 운영 현황 표시

| 구분 | 조건 | 비고 |
|------|------|------|
| 도착 (Arrivals) | 오늘 체크인 예정 서브예약 수 | 체크인 완료 건수를 별도 표시 |
| 재실 (In-House) | 현재 투숙 중인 서브예약 수 | |
| 출발 (Departures) | 오늘 체크아웃 예정 서브예약 수 | 체크아웃 완료 건수를 별도 표시 |

## 부록 A. 상태 전이 다이어그램

### A.1 예약 상태 전이

```
                 ┌──────────┐
            ┌───>│ CANCELED │
            │    └──────────┘
            │
┌──────────┐│   ┌──────────┐    ┌─────────────┐
│ RESERVED ├┼──>│ INHOUSE  ├───>│ CHECKED_OUT │
└────┬─────┘│   └──────────┘    └─────────────┘
     │      │
     │      └──>┌──────────┐
     └─────────>│ NO_SHOW  │
                └──────────┘
```

### A.2 HK 태스크 상태 전이

```
┌─────────┐    ┌─────────────┐    ┌───────────┐    ┌───────────┐
│ PENDING ├───>│ IN_PROGRESS ├───>│ COMPLETED ├───>│ INSPECTED │
└────┬────┘    └──┬───┬──────┘    └───────────┘    └───────────┘
     │            │   │
     │            │   └───>┌────────┐
     │            │        │ PAUSED ├──┐
     │            │        └────────┘  │ (back to IN_PROGRESS)
     │            └────────────────────┘
     │
     └──────────────────>┌───────────┐
     (from any non-done) │ CANCELLED │
                         └───────────┘
```

### A.3 결제 상태 전이

```
┌────────┐    ┌─────────┐    ┌──────┐
│ UNPAID ├───>│ PARTIAL ├───>│ PAID │
└────────┘    └─────────┘    └──┬───┘
                                │ (adjustment)
                                v
                           ┌──────────┐
                           │ OVERPAID │
                           └──────────┘

취소/노쇼 시: 어떤 상태에서든 → REFUNDED (명시적)
```

### A.4 객실 상태 조합

```
FO Status + HK Status 조합:

VACANT + CLEAN     = VC (입실 가능)
VACANT + INSPECTED = VI (검수 완료, 입실 가능)
VACANT + DIRTY     = VD (청소 필요)
VACANT + PICKUP    = VP (검수 대기)
VACANT + OOO       = 판매 불가
VACANT + OOS       = 사용 불가
OCCUPIED + CLEAN   = OC (재실 청소 완료)
OCCUPIED + DIRTY   = OD (재실 청소 필요)
OCCUPIED + DND     = 방해 금지
```
