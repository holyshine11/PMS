# 역할
당신은 호텔 PMS 및 부킹엔진 설계/구현 경험이 있는 시니어 소프트웨어 아키텍트이자 실무형 테크 리드입니다.  
목표는 내가 현재 개발 중인 PMS와 연동되는 **예약 접수용 부킹엔진(Booking Engine) 프로토타입**을 안정적으로 설계하고,  
실제 PG 연동 이전 단계에서 **가상 예약 화면 + PMS 예약 적재 기능**을 우선 구현할 수 있도록  
작업 범위, 설계, 태스크 분해, 영향도, 병렬 작업 계획까지 실무적으로 정리하는 것입니다.

# 프로젝트 배경
내가 개발 중인 PMS가 있고,  
향후 **A라는 호텔 사이트에서 예약이 발생하면 PMS 예약 목록으로 정상 적재되는 구조**를 만들고 싶다.

다만 현재는 아래 제약이 있다.

1. 호텔 사이트 전체를 지금 당장 만들 수는 없다.
2. 먼저 테스트를 위한 **가상의 예약 화면(프론트)** 을 만들어야 한다.
3. 이 화면은 PMS가 가지고 있는 예약 관련 기준정보를 받아와서 예약을 구성해야 한다.
4. 기준정보 예시:
   - 프로퍼티(Property)
   - 층/호수
   - 객실 타입 / 객실 정보
   - 레이트(rate)
   - 예약 가능 조건
   - 투숙 인원
   - 체크인/체크아웃 날짜
   - 기타 예약에 필요한 메타 정보
5. 사용자가 예약정보를 입력하고 결제 버튼을 클릭하면,
   **실제 PG 연동 전이라도 예약 성공 처리 후 PMS 예약 목록에 적재되는 기능**을 먼저 만들어야 한다.
6. 결제는 추후 붙일 예정이며 계획은 다음과 같다.
   - 한국: KICC PG
   - 해외: PayPal PG
7. 현재 단계에서는 **실결제 연동이 아닌 Mock/가상 승인 흐름**으로 먼저 예약 적재 기능을 안정화하는 것이 목적이다.
8. 충분히 안정화된 이후 PG 연동을 추가할 계획이다.

# 반드시 지켜야 할 작업 원칙
아래 원칙을 반드시 준수하라.

## 1. 한 번에 구현하지 말 것
작업 분량이 많기 때문에 한 번에 전부 진행하지 말고,  
**영향 범위가 격리된 단위 태스크로 잘게 분리**하라.

## 2. 먼저 구현하지 말고 분석부터 할 것
곧바로 코드부터 작성하지 말고 아래 순서로 진행하라.

1. 요구사항 재정리
2. 누락된 요구사항 및 리스크 도출
3. 전체 아키텍처 제안
4. 데이터 흐름 설계
5. API 및 상태 정의
6. 태스크 분해
7. 병렬 처리 가능한 작업 묶음 정의
8. 충돌 방지 전략 정의
9. 1차 작업계획안 제시
10. 그 다음에만 구현 착수

## 3. 내가 제안한 내용에서 부족한 부분을 반드시 보강할 것
단순 요약이 아니라,  
내가 놓쳤을 가능성이 높은 항목들을 실무적으로 추가 보완하라.

예를 들면 아래 같은 관점까지 포함해야 한다.
- 예약 상태값 설계
- 임시예약 / 예약확정 구분 여부
- 객실 재고 차감 시점
- 동시성 이슈
- 중복 예약 방지
- 가격 확정 시점
- 객실/레이트 유효성 검증
- 체크인/체크아웃 validation
- 취소/변경 고려 필요성
- 결제 전/후 상태 분리
- PG 연동 확장 가능 구조
- 실패 복구
- 로깅/감사 추적
- 테스트 전략
- 향후 실제 호텔 사이트 연동을 위한 확장성

## 4. PG는 지금 붙이지 말고, 나중에 붙이기 쉬운 구조로 설계할 것
현재 단계에서는 PG 연동 대신  
**결제 버튼 클릭 → mock 승인 → PMS 예약 생성** 흐름을 구현하되,  
나중에 아래 PG를 쉽게 붙일 수 있도록 인터페이스/어댑터 구조를 고려하라.

- KICC
- PayPal

즉, 지금은 MockPaymentGateway 같은 구조를 쓰고,  
추후 KICCPaymentGateway / PayPalPaymentGateway 로 대체 가능해야 한다.

## 5. 병렬 처리 계획을 제시할 것
작업 효율을 위해 Claude 팀이 병렬로 처리할 수 있도록  
**중첩/충돌이 적은 작업 묶음(workstream)** 으로 나누어라.

단, 병렬화 시 아래를 반드시 제시하라.
- 어떤 작업끼리 병렬 가능한지
- 어떤 선행조건이 필요한지
- 충돌 가능 파일/모듈
- 인터페이스 계약(API contract / DTO / schema)을 먼저 무엇으로 고정해야 하는지
- 통합 시점 체크포인트

## 6. 불확실해도 멈추지 말고 합리적 가정을 세워 진행할 것
정보가 부족해도 작업을 멈추지 말고,  
합리적인 가정을 세워 분석과 설계를 계속 진행하라.  
다만 마지막에 **“가정한 사항”** 을 별도로 정리하라.

## 7. 결과물은 실무 문서처럼 구조적으로 작성할 것
답변은 추상적으로 하지 말고,  
실제 개발 착수 전에 PM/아키텍트/개발리더가 검토 가능한 수준으로 작성하라.

각 Phase마다 아래를 작성하라.

- 트랙 B: 예약 도메인/DB/API 설계
- 트랙 C: Mock 결제 추상화 계층

각 트랙별로 아래를 작성하라.
- 담당 범위
- 입력/출력 계약
- 다른 트랙과의 의존성
- 충돌 가능성
- 병합 순서

# 기술적 기대사항
다음은 가능한 한 실무적으로 고려하라.

1. Mock 결제 구조는 실제 PG 연동으로 쉽게 치환 가능해야 한다.
2. PMS 예약 생성은 트랜잭션 처리 관점에서 설명하라.
3. 예약 생성 시 필요한 스냅샷 저장 여부(당시 객실명/요금명/금액)를 검토하라.
4. 예약은 외부 사이트 유입 채널 개념을 고려하라. 예: DIRECT_WEB
5. 실제 호텔 사이트 대신 사용할 테스트용 예약 화면은 너무 단순하지 않게,  
   실제 예약 플로우 검증이 가능하도록 설계하라.
6. 향후 호텔별 커스터마이징 가능성도 고려하라.
7. 환율/통화/내국인·외국인 구분 등이 필요할 가능성도 검토하라.
8. 최소한의 관리자 확인 포인트(예약 생성 로그, 결제 시뮬레이션 결과, 실패 원인)를 제안하라.

# 매우 중요한 추가 지시
- 내가 바로 구현에 들어갈 수 있도록 최대한 구체적으로 작성하라.
- 추상적인 원론 설명보다 **실제 설계/개발 문서 수준**으로 작성하라.
- 가능하면 표 형식 또는 계층형 구조로 보기 좋게 정리하라.
- 한 번에 끝내려 하지 말고, 먼저 **분석/설계/계획**을 완성하라.
- 구현이 필요한 경우에도 우선순위가 높은 최소 범위부터 제안하라.
- 병렬 처리를 고려하되, 중첩과 충돌을 최소화하라.
- 불명확한 부분은 멈추지 말고 합리적으로 가정해서 설계하라.


추가 지시:

1. 답변은 PM, 아키텍트, 백엔드 개발자, 프론트 개발자, QA가 함께 보는 설계 문서 수준으로 작성할 것.
2. “좋을 수 있다”, “고려할 수 있다” 같은 표현만 남발하지 말고, 가능한 경우 권장안을 명확히 결정해서 제시할 것.
3. 모든 제안은 “왜 그렇게 설계하는지” 이유를 짧게 덧붙일 것.
4. API/상태/도메인/태스크는 반드시 상호 정합성이 맞아야 한다.
5. 병렬 처리 계획은 실제 여러 작업자가 동시에 진행 가능한 수준으로 파일/모듈 경계까지 제안할 것.
6. 마지막에는 반드시 “권장 구현 순서 Top 10”을 체크리스트 형태로 정리할 것.
7. 구현 착수 전에 반드시 “지금 범위에서 제외할 것”도 명확히 선언할 것.
