# 올라(Hola) PMS 시스템 개발 계획서 v1.1

> **최종 수정일**: 2026-03-10 | **문서 버전**: v1.1 (개발 진행 현황 반영)

## 1. 프로젝트 개요

### 1.1 비전

국내 3~5성급 중견 호텔 대상, Oracle Opera Cloud PMS의 한국형 대안. 호스팅·운영·지원 포함 구독형으로 도입 장벽을 낮추고, API-First 아키텍처로 확장성을 확보한다.

### 1.2 핵심 수치

| 구분 | 내용 |
|------|------|
| **글로벌 PMS 시장** | $2.9~3.8B (2025), CAGR 9~12% |
| **타겟 시장** | 국내 3~5성급 중소형 체인 호텔 300개소+ |
| **가격 경쟁력** | 객실당 3,000~9,000원/월

### 1.3 시장 진입 전략 (Go-to-Market 5단계)

| Phase | 목표 | 내용 |
|-------|------|------|
| **1. 레퍼런스 확보** | 14개 프로퍼티 실증 | 신라호텔(9) + 조선호텔(5) 무상 구축 |
| **2. 유상 파일럿** | Basic 플랜 전환 | 제주도 소형 호텔 3~5곳, 부산/강릉 리조트 확장 |
| **3. 국내 확장** | 50+ 프로퍼티 | 라한, 콘레드, 글래드 체인 공략 (Standard 플랜) |
| **4. 해외 진출** | 방콕 파일럿 | 태국 중형 호텔, Agoda/Booking.com 연동 |
| **5. 글로벌 확장** | 연 매출 100억 | 동남아/일본 주요 도시, Enterprise 플랜 |

### 1.4 경쟁사 비교

| 구분 | Oracle Opera | 산하정보기술(야놀자) | Cloudbeds | **Hola(올라)** |

### 2.2 모듈 간 API 통신 규칙

```
[모듈 간 통신 원칙]

2. 비동기 처리: RabbitMQ Event (예약 생성 → 채널매니저 재고 동기화)
3. 데이터 참조: 각 모듈은 자신의 도메인 테이블만 직접 접근
4. 타 모듈 데이터 필요 시: API 호출 또는 Event 수신으로 해결
```

### 3.2 시스템 아키텍처 (Modular Monolith)

```
[CLIENT] PMS Admin (Thymeleaf SSR) │ Booking Web │ Mobile App ─── CDN (CloudFront)
                              │
[GATEWAY] Spring Cloud Gateway (JWT Auth / Rate Limit / Tenant Resolution)
                              │
[MODULAR MONOLITH ─── Spring Boot 3.2 ────────────────────────────────────]
│ M01 hotel │ M02 room  │ M03 rate    │ M07 reservation │ M10 frontdesk  │
│ M04 hk    │ M05 channel│ M06 pos    │ M08 booking     │ M09 billing    │
│ M11 ai    │ M12 report │ common     │ notification    │                │
│──────────── Event Bus (RabbitMQ) ─────────────────────────────────────── │
[DATA] PostgreSQL 16 (Schema-per-Tenant) │ Redis 7 │ S3 │ ES 8.x
```

### 3.3 멀티테넌시 전략

**Schema-per-Tenant** 채택: PostgreSQL 스키마 단위로 테넌트 격리.

- `hola_common`: 호텔/프로퍼티 마스터, 시스템 코드, 관리자 계정
- `hola_{hotel}_{property}`: 객실, 레이트, 예약, 정산 등 프로퍼티별 데이터
- API Gateway에서 `X-Tenant-ID` 헤더로 테넌트 결정 → ThreadLocal 기반 스키마 전환

### 3.4 Modular Monolith → MSA 전환 전략

| 단계 | 시기 | 구조 | 이유 |
|------|------|------|------|
| **Phase 1~3** | 2026.03~2027.02 | Modular Monolith | 5인 팀으로 MSA 운영 부담 회피 |
| **Phase 4+** | 2027.03~ | 점진적 MSA 분리 | 트래픽 증가 시 채널매니저, 알림 순으로 분리 |

## 5. API 설계

### 5.1 설계 원칙

- RESTful 리소스 중심, 버전 관리 `/api/v1/...`
- JWT Bearer Token + `X-Tenant-ID` 헤더
- 통일된 JSON 응답: `{success, code, message, data, pagination, timestamp}`
- 에러 코드 체계: `HOLA-XXXX`
- 페이징: `page`, `size`, `sort` 표준화

### 13.3 Claude Code AI 팀 활용 전략

12개 모듈을 소규모 팀으로 구축하기 위해 Claude Code를 **6번째 팀원**으로 운영한다.

#### A. 모듈별 개발 사이클 (Claude Code 통합)

```
[1턴] PM: 설계서 기반 기능 명세 작성 (CLAUDE.md + 설계 PPT 참조)
         ↓
[2턴] Claude Code: Backend 코드 생성 (Entity → Repository → Service → Controller)
         ↓
[3턴] 개발자: 코드 리뷰 + 비즈니스 로직 보강 + 예외 케이스 처리
         ↓
[4턴] Claude Code: 단위/통합 테스트 생성 + Flyway 마이그레이션 스크립트
         ↓
[5턴] Claude Code: Frontend 페이지 생성 (Ant Design 기반 리스트/폼/모달)
         ↓
[6턴] 개발자: UI 리뷰 + API 연동 검증 + E2E 시나리오 테스트
```

> 핵심 원칙: **1턴 1기능**, 매 턴 결과 확인 후 다음 진행. 30페이지 스펙 한 번에 전달 금지.

#### B. 프로젝트 컨텍스트 관리

| 파일 | 용도 | 관리 주기 |
|------|------|-----------|
| `CLAUDE.md` (프로젝트 루트) | 기술 스택, 코드 컨벤션, 패턴 가이드 | 프로젝트 초기 1회 + 변경 시 |
| `docs/modules/M{XX}_spec.md` | 모듈별 설계 명세 (API, 테이블, 화면) | 모듈 착수 시 |
| `docs/conventions.md` | 네이밍, 패키지 구조, 응답 형식 규칙 | Phase 0 확정 |

**CLAUDE.md 핵심 항목:**
- 패키지 구조: `com.hola.{module}.controller/service/repository/dto/entity`
- 네이밍: camelCase 변수, PascalCase 클래스, UPPER_SNAKE 상수
- 응답 형식: `HolaResponse<T>` 공통 래퍼
- 예외 처리: `HolaException` + `HOLA-XXXX` 에러 코드
- 테넌트: `@TenantRequired` 어노테이션으로 자동 스키마 전환
- 참조 패턴: "M01 호텔관리 코드를 참고하여 동일한 패턴으로 구현하라"

#### C. Claude Code 모듈별 활용 맵

| 단계 | Claude Code 역할 | 사람 역할 | 생산성 배수 |
|------|------------------|-----------|:----------:|
| **ERD → Entity** | DDL 기반 JPA Entity + Auditing 자동 생성 | 관계/제약조건 검증 | 5~8x |
| **CRUD API** | Controller/Service/Repository 패턴 코드 | 비즈니스 규칙 보강 | 4~6x |
| **복잡 로직** | 초안 코드 + 테스트 케이스 생성 | 레이트 계산, 상태전이 등 핵심 로직 검증 | 2~3x |
| **Frontend** | Ant Design 리스트/폼/모달 페이지 | UX 흐름 검증, 권한 체크 | 3~4x |
| **테스트** | JUnit + Testcontainers 테스트 자동 생성 | 경계값/예외 시나리오 추가 | 3~5x |
| **DB 마이그레이션** | Flyway 스크립트 + 인덱스 생성 | 성능 영향 검토 | 4~5x |
| **코드 리뷰** | 보안/성능/컨벤션 자동 체크 | 아키텍처 판단 | 2~3x |
| **문서화** | API 문서 (OpenAPI/Swagger) 자동 생성 | 비즈니스 설명 보완 | 5x |

#### E. 10턴 초과 시 컨텍스트 관리

- 10턴 이상 연속 작업 → `/compact` 실행 (핵심 결정사항 백업 후)
- 모듈 전환 시 → 새 세션 시작 (CLAUDE.md가 컨텍스트 복원)
- 복잡한 디버깅/아키텍처 결정 → `think harder` 모드 활용
- 5개+ 파일 수정 → Task 리스트로 진행 상황 추적
