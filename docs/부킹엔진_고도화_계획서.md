## 3. 고도화 계획

### Phase 1: 핵심 API 보강 (부킹 플로우 완성)

부킹엔진의 완전한 예약 플로우를 위한 핵심 API 보강.

#### 3.1.1 신규 API (4개)

**A. 캘린더 API** (`GET /api/v1/booking/properties/{code}/calendar`)
- 판매 가능 날짜 목록 반환
- 체크인 가능일 / 체크아웃 가능일 분리 모드 (`?type=checkin|checkout|all`)
- 산하 API 2.2 + 2.3 통합

**B. 패키지(레이트) 목록 API** (`GET /api/v1/booking/properties/{code}/rate-plans`)
- 조건: 체크인/아웃, 객실수, 성인/아동, 프로모션코드
- 응답: 패키지명, 이미지, 상품구성, 최저가, 정렬순서
- 산하 API 2.4 대응

**C. 패키지 상세 API** (`GET /api/v1/booking/properties/{code}/rate-plans/{id}`)
- 요금 상세, 취소정책(단계별 %), 노쇼정책, 상품구성
- HTML 약관 내용 포함
- 산하 API 2.5 대응

**D. 객실 상세 API** (`GET /api/v1/booking/properties/{code}/rooms/{id}`)
- 객실 시설, 사진 목록, 최대인원(성인/아동), 침대타입
- 어메니티 타입코드, CMS 연결 ID
- 산하 API 2.8 대응

#### 3.1.2 기존 API 보강 (4개)

**A. availability 보강**
- 일별 요금(daily_rate) 포함 옵션 추가 (`?includeRates=true`)
- 캘린더 모드 지원 (날짜 목록만 반환)
- 산하 API 2.7 완전 대응

**B. price-check 보강**
- 패키지별 그룹핑 응답
- 최저가 패키지 표시
- 성인 추가 요금(add_adult_amt) 계산 포함
- 산하 API 2.6 완전 대응

**C. confirmation 보강**
- 호텔 정보 (주소, 전화, 체크인/아웃 시간)
- 세금/봉사료 상세
- 결제 상세 (카드 유형, 마스킹된 카드번호)
- 취소 정책 정보
- 산하 API 2.12 완전 대응

**D. cancel 보강**
- 취소 사유 코드 추가 (`cancelReasonCode`)
- 객실별 개별 취소 지원 (rsvn_room_list)
- 취소 수수료 단계별 상세 반환
- 산하 API 2.15 완전 대응

#### 3.1.3 인프라 보강

**A. API Key 인증 체계**
- `BookingApiKey` 엔티티 (벤더별 API Key 관리)
- `BookingApiKeyFilter` (Security Filter Chain에 추가)
- 헤더: `API-KEY`, `VENDOR-ID`
- Admin UI에서 벤더/키 관리

**B. 다중 통화 지원**
- 지원 통화: **KRW, USD, JPY, CNY**
- `currency` 파라미터 추가 (기본값: KRW)
- 환율 테이블 (`exchange_rate`) + 관리 API
- 요금 응답에 통화코드 포함

**C. 부킹 전용 응답 래퍼**
- `BookingResponse<T>` (산하 형식 호환)
- 성공: `{ "result": { ... } }`
- 실패: `{ "result": { "RESULT_YN": "N", "RESULT_MESSAGE": "..." } }`

### Phase 2: 부가 기능 (사용자 경험 강화)

#### 3.2.1 신규 API (5개)

**A. 객실-패키지 역방향 조회** (`GET .../rooms/{id}/rate-plans`)
- 특정 객실에 적용 가능한 패키지 목록
- 일별 요금 포함
- 산하 API 2.9 대응

**B. 옵션 조회** (`GET .../rooms/{roomId}/rate-plans/{rateId}/options`)
- 조식, 스파 등 부가 서비스
- 옵션별 요금, 최대 수량, 제공 유형(BF_NIG 등)
- 산하 API 2.10 대응

**C. 마이페이지 예약 목록** (`GET /api/v1/booking/my/reservations`)
- 조회 타입: Book(예약), Old(이전 내역), Cancel(취소/노쇼)
- 게스트 인증: 이메일+전화 or 예약번호
- 산하 API 2.11 대응

**D. 예약 수정** (`PUT /api/v1/booking/reservations/{no}`)
- 투숙객 정보 변경 (이름, 이메일, 연락처, 국가코드)
- 산하 API 2.14 대응

**E. 프로모션 코드 검증** (`POST .../validate-promotion`)
- 프로모션 코드 입력 → 유효성 확인 + 프로모션 ID 반환
- 산하 API 2.17 대응

#### 3.2.2 기존 보강

**A. 예약-패키지-객실 통합 정보** (`GET .../reservations/{no}/rate-info`)
- 예약 기반 패키지+객실+취소정책 통합 조회
- 산하 API 2.16 대응

### Phase 3: 외부 연동 표준화 (채널 연동)

#### 3.3.1 신규

**A. 카드 BIN 검사** (`POST /api/v1/booking/card-check`)
- 카드번호 앞 6자리로 유형 판별
- AX(Amex), DR(Discover), JC(JCB), MS(Master), VI(Visa), UN(UnionPay), CD(Local)
- 산하 API 2.18 대응

**B. 외부 코드 매핑 테이블**
- `booking_code_mapping` 테이블
- 벤더별 외부코드 ↔ Hola 내부 ID 매핑
- 호텔, 객실, 패키지 각각 매핑

**C. 채널 코드 체계**
- 예약 소스 코드 관리 (WINGS_B2C, BE4PC 등)
- 기존 ReservationChannel 엔티티와 연동

## 5. 기존 코드 보완사항

### 5.1 테스트 코드 부재

현재 전체 프로젝트에 테스트 코드 없음. 최소한 부킹엔진 핵심 서비스에 대해:
- `BookingService` 단위 테스트
- `PriceCalculationService` 단위 테스트
- 부킹 API 통합 테스트 (MockMvc)

### 5.2 노쇼 자동 처리

- 엔티티만 있고 자동화 없음
- 스케줄러로 체크인 시간 이후 미도착 예약 자동 NO_SHOW 처리
- 노쇼 수수료 자동 계산

### 5.3 API 문서 (Swagger/OpenAPI)

- 부킹 API는 외부 연동용이므로 API 문서 필수
- springdoc-openapi 적용 권장
- 벤더에게 제공할 연동 가이드 자동 생성

### 5.4 Rate Limiting

- 부킹 API 호출 제한 (벤더별 분당/시간당)
- Bucket4j 또는 Spring Cloud Gateway 적용

### 5.5 Redis 설정 정리

- 현재 Redis 의존성 및 설정은 있으나 실사용 없음
- `build.gradle`에서 Redis 의존성 제거
- `application.yml`에서 Redis 설정 제거
- 추후 캐싱 필요 시 재도입
